# Core principles

The rules of DSK itself. Fixed. Every concept in the other files either follows from these or refines them.

1. **Source of truth is authoritative and the only origin.** DSK reads from the declared source of truth; it never overrides it, and it never invents new layouts, content types, or brand elements on its own. All extensions to the system originate in the source of truth and flow into DSK via snapshotting, never the other way. When a user requests something that would require a new layout or content type, the agent directs them to update the source of truth first; never creates it inside DSK.
2. **Everything DSK generates is regenerable.** Any artifact in the company zone (welcome page, layouts page, examples, content gallery, extracted materials) is a build artifact. It can be deleted at any time and rebuilt from source of truth plus kernel rules. This is what makes DSK low stakes and portable.
3. **Strict kernel/company separation.** The kernel zone is identical across every company. The company zone holds source of truth and company specific content. They only meet at a small config surface.
4. **Respect, then stretch.** The agent's first move is to match the source of truth exactly. If user intent does not fit, it takes the smallest brand consistent liberty. Larger deviations require explicit confirmation.
5. **Degrees of freedom are tunable per company.** Some companies want near zero liberty, others allow more. This setting affects both runtime behavior and what the library pages show.
6. **Vendor neutral by construction.** DSK assumes only "a folder of files an agent can read." No platform specific API calls.
7. **Rules are behavior level, not implementation.** DSK is designed for the agentic AI age, where the right move is to set high level direction and constraints and let agents work within them, not to dictate how. The kernel describes what each artifact must convey, not how to build it. Two runs may produce different rendered output while both satisfying the same contract. Analogous to BDD: behaviors are the contract; implementation is left to the agent.
8. **Destructive changes require explicit consent.** The agent never deletes or makes large structural changes to the company zone silently. When source of truth changes would remove a layout, retire an example, or significantly alter existing materials, the agent stops, explains what would change and why, describes the consequence, and asks the user to confirm. Default behavior is to preserve. Goal: stay aligned with source of truth, but never destroy or rewrite by accident.
9. **Composable with upstream AI processes.** Content for slide generation can arrive plain (raw) or annotated (with metadata produced by upstream systems such as a company knowledge base, a content generator, or a research agent). DSK is metadata-aware: when conventional metadata keys are present (style hints, layout hints, precedents), it uses them to inform layout selection and rendering. DSK does not define what produces the metadata; it only consumes it. This keeps DSK at the design system layer while letting companies build sophisticated content pipelines around it. Plain mode is the default; annotated mode adds intelligence when available.
10. **Faithful translation, additive expressivity.** The output medium is web technologies, not PowerPoint. The floor is non-negotiable: translation must never be perceived as a downgrade from the source. The ceiling is opt-in: web's extra capabilities (interactivity, live data, responsive layout, custom typography) are available when invited by the user or by annotated metadata, not imposed by default.
11. **Snapshot stage is host-portable.** The snapshot pipeline (source of truth → snapshot.json + assets/) must run in any skill-compatible agent runtime (Claude Code, Codex, equivalents). This is the testable boundary; downstream stages (Build, Compose, Sync) live in the AI Design Tool and are exercised there. Reason: the snapshot is DSK's distinctive output, and verifying it must not require burning AI Design Tool tokens.
